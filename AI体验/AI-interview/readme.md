# RAG 架构与检索流程

## 为什么引入父子索引?

- 为了解决检索精度与上下文完整性的矛盾
- 传统RAG切割文档：
  1. 切大了上下文完整，但是大量的无关信息导致向量匹配度低。
  2. 切小了向量匹配度高，但是上下文碎片化严重，AI会断章取义。
- 思路：
  父文档保证较为完整的上下文，子文档根据父文档进行切分，子文档只用来索引。当子文档被匹配到时直接返回父文档（检索与召回机制）

## 为什么在检索阶段引入BM25?

- 为了解决“向量检索”在关键词匹配精准度上的问题，通过混合检索提升整体召回率和鲁棒性
- 向量检索：能够检索语义相关的词语（如：汽车和机动车相关）。但是对于一些领域外的关键词，容易返回错误答案。因为稀有词在向量维度中容易被稀释。
- BM25：基于关键词的检索算法，擅长检索特定的关键词。
- 混合检索：它俩混合一起，向量检索负责懂意思，BM25负责关键词精确度

## (追问)BM25和向量检索是怎样组合的?比例是如何设置的?

1. BM25和向量检索的组合方式
   RRF分数 = 1/(k + BM25排名) + 1/(k + 向量排名)（k通常取60）

- 加权融合（需归一化）
  - 公式  
    归一化分数 = (原始分数 - 最小值) / (最大值 - 最小值)
    最终分数 = α × 归一化向量分数 + (1-α) × 归一化BM25分数
  - 限制
    归一化对单次查询结果分布敏感，若存在异常高分值，可能导致其他结果分数被压缩
- RRF(排名融合)
  - 公式
    RRF分数 = 1/(k + BM25排名) + 1/(k + 向量排名)（k通常取60）
  - 优势
    无需归一化：天然规避分数量纲问题。
    鲁棒性强：不受异常值影响，默认参数（k=60）在多数场景下效果稳定，无需频繁调参
- 级联检索（分阶段优化）
  - 流程：
    BM25初筛（召回Top 1000）→ 向量精排（Top 100）→ Rerank重排序（Top 5~10）。
  - 适用场景：
    大规模数据（如亿级文档）下平衡效率与精度，避免向量检索全量计算开销

2. 比例如何设置：需要分场景
   - 通用场景（日常问答、模糊查询）：
     向量权重 0.6~0.7，BM25权重 0.3~0.4。
     侧重语义理解，兼顾关键词匹配。
   - 专业术语密集场景（法律条文、医疗术语、错误码）：
     BM25权重 0.5~0.6，向量权重 0.4~0.5。
     例如：搜索"XJ-2397规则"或"user_status字段"时，BM25的精确匹配能力更关键。
   - 短查询/精确匹配场景（<10字）：
     BM25权重可提升至 0.7，向量权重降至 0.3。
     短查询语义信息少，向量检索易失效

## (追问)整体检索流程是怎样的?从 query（提问） 到最终上下文的完整流程是什么?

### 整体流程：

1.  预处理阶段
    - 意图识别：通过轻量模型或规则识别用户的问题类型（如事实查询、操作指导），避免模糊表述产生检索偏差。
    - 查询扩展：
      （1） 同义词扩写（如：心梗 -> 心肌梗塞）
      （2） HyDE 技术：用 LLM 生成假设性答案作为检索查询，提升语义匹配深度。
    - 元数据过滤：根据用户权限、身份动态过滤知识库范围。

2.  多路检索
    - 混合检索
      - 向量检索
      - BM25
    - 结果融合
      使用RRF对检索结果按排名融合
3.  结果精排与上下文优化
    - Rerank 重排序
      （1）用Cross-Encoder精排：用 BERT 类模型对 Query（提问） 与每个候选文档进行细粒度相关性打分。
      （2）根据时效性、来源权威性等人工规则调整分数。
    - 上下文压缩与扩展
      （1）用LLM压缩冗余内容，保留核心事实。
      （2）父子索引，召回父文档补充上下文。
      （3）添加邻近文本块，保证逻辑连贯性。
4.  生成前上下文组装
    - 动态排序与截断
      （1）按Rerank 分数倒序排列，保证模型token数 < 模型上下文窗口容量。
      （2）保留高分数内容，丢弃分数较低的内容，降低幻觉。
    - 结构化 Prompt
      （1）将结果拼接成结构化 Prompt
      （2）明确文档来源
      （3）约束指令让模型依赖检索内容

## (追问)检索阶段有没有做 rerank ? 使用的是什么方式?

### rerank

- 是对初筛检索结果进行精细化二次打分排序的关键步骤，核心是用更精准的模型替代向量相似度粗排序，将真正相关的文档排到最前。其本质是牺牲少量计算时间换取显著精度提升

1. Cross-Encoder 精排模型
   将问题与候选文档拼接为单输入，使用完整transformer模型计算细粒度相关性分数

   ```
    from sentence_transformers import CrossEncoder
    reranker = CrossEncoder("BAAI/bge-reranker-v2-m3", max_length=8192)
    pairs = [[query, doc] for doc in candidate_docs]  # 拼接Query-Document对
    scores = reranker.predict(pairs)                # 获取相关性分数
    reranked_docs = [doc for _, doc in sorted(zip(scores, candidate_docs), reverse=True)]
   ```

2. LLM-Based 重排（高成本方案）
   - Prompt指令排序：让LLM直接对候选文档列表打分（如“按相关性从高到低排序以下文档”）
   - RankGPT：用滑动窗口Listwise策略，让LLM逐组对比文档并输出排序
     对精度要求极高且可接受高延迟的离线任务（如法律条文检索），线上服务慎用（GPT-4单次推理超500ms）
3. 轻量级特征融合重排
   将 BM25分数、向量相似度、时效性、来源权威性 等多维度信号加权融合：最终分数 = 0.4×Rerank分数 + 0.3×BM25分数 + 0.2×时效权重 + 0.1×来源权重。

## rerank 后一般返回几个块?

### 影响因素

1. 业务场景需求
   - 高精度场景（如金融、医疗）返回5-10个块，确保关键信息不遗漏
   - 通用问答（如客服、日常对话）返回3-5个块，避免冗余信息干扰生成
2. 模型上下文窗口token数
3. 性能与成本
   - rerank模型是高精度计算、打分，块越多，成本越高

## rerank后的topK截断是怎么做的?

1.  根据大模型上下文长度和Query实际占用Token动态计算剩余空间

```python
final_context = [doc for doc, score in rerank_results[:top_k]]  # 直接切片取前K个


<!-- topK 动态计算 -->
def dynamic_topk(query, rerank_results, max_tokens=4000):
    used_tokens = len(tokenizer.encode(query))
    top_k = 0
    for doc, score in rerank_results:
        doc_tokens = len(tokenizer.encode(doc))
        if used_tokens + doc_tokens > max_tokens * 0.8:  # 保留20%缓冲
            break
        used_tokens += doc_tokens
        top_k += 1
    return max(1, top_k)  # 至少保留1个文档
```

2. 基于分数落差的“断崖截断”.
   - 例如：若第3名分数（0.88）与第4名（0.73）差距 > 阈值（0.15），则在第3名后截断

```python
def cliff_topk(rerank_results, min_drop=0.15, min_k=3):
    scores = [score for _, score in rerank_results]
    for i in range(1, len(scores)):
        if scores[i-1] - scores[i] > min_drop:  # 分数突降处截断
            return max(min_k, i)  # 至少保留min_k个
    return min_k
```

## (追问)如果上下文长度不够或过长,你是怎么处理的?

### 上下文过长

1. 动态压缩与摘要
   - 当对话历史超过70%时，通过独立的API压缩成结构化摘要并保留最近几轮对话。
   - 增量摘要：将旧摘要与新增对话融合成新的摘要
2. 智能截断与滑动窗口
   - 滑动窗口：保留最近10轮对话，保留系统关键提示词，用户身份信息，丢弃非必要信息。
   - 智能截断：根据语义权重对历史对话进行打分（如：必要，务必），截断评分低的。

### 上下文过短

1. 使用轻量模型解析用户输入参数的缺失，针对高频场景预设补全逻辑。
   如：用户“查询订单”，模型回答：您要查询的订单号或手机号？
2. 场景化示例引导
3. 历史行为预填充：根据用户输入的历史行为，对用户输入进行补全。
4. 多模态补全
   对缺失信息通过其他手段获取，如：缺失地理位置信息时，直接调用地图API获取，显示：“正在获取您的地理位置信息...”

# 上下文工程与记忆机制

## 讲一下上下文工程是怎么设计的。

1. 动态信息筛选与优先级管理
   - 设置Token预算：为系统指令、检索结果、对话历史、工具调用等模块做Token上限，超过自动触发压缩逻辑。
   - 动态打分：对历史消息按“时效性”，“重要性”，“相关性”等打分，筛选高分片段。
   - 位置优化：利用模型对上下文首尾注意力更高的特性，将核心约束信息放在首尾。
2. 结构化信息与协议化
   - 模块化上下文：将历史消息拆分为独立区块，如：<system>-角色指令，<memory>-关键记忆，<rag>-检索结果
   - 采用MCP：给不同来源信息统一协议
   - 语义化压缩：将原始数据转化为模型更容易看得懂的信息，如：数据库错误日志 ->压缩为：404 | 资源未找到
3. 智能化压缩与状态维护
   - 摘要：将历史对话压缩为摘要，新增对话与旧摘要合成新的摘要
   - 分层记忆
     - 短期记忆：保留最近几轮对话，超出部分保留为摘要
     - 长期记忆：保留关键事实到向量数据库，使用时只召回几条，避免一次性注入全部历史。
   - 任务状态显示化：显示当前任务状态，防止模型忘记而重复提问。如：步骤2/5，等待用户确认。
4. 实时监控和降级机制
   - Token用量实时监控：在请求链路中注入实时监控，并记录token用量，超过阈值则触发降级（如缩减RAG片段）。
   - 动态降级：当上下文接近窗口上限时，丢弃低优先级内容
   - 模型回答质量降低：回溯上下文是否存在上下文冲突（新旧文档矛盾）或信息过载。
   - AB测试验证：对比不同上下文策略的准确率、延迟、成本，持续优化筛选规则（比如发现8000 tokens上下文回答最优）。

## (追问)上下文拼接的结构是怎样的?

1. 系统指令层
   - 定义模型的指令、规范、角色

2. 动态知识库/记忆层
   - 定义外部检索到的文档、数据库存储的摘要

3. 对话历史层
   - 用户与模型的多轮对话，一般会压缩

4. 当前任务/工具层
   - 当前用户的问题、可用的工具列表、工具执行的结果

## 记忆机制是怎么做的?

### 短期记忆

- 滑动窗口：每次保留最近的N轮对话
- 摘要：将历史对话压缩为摘要，新增对话与旧摘要合成新的摘要；容易丢失细节
- Token截断：超出Token限制时截断旧内容

### 长期记忆

- 向量存储：将用户的偏好和关键事实向量化存储。、
  （1） 在对话结束后异步调用大模型将对话中用户的偏好和关键事实向量化存储起来，转化为结构化的用户画像，存入向量库。
- 检索增强：发起提问时，系统先通过语义检索与当前最相关的历史片段，注入到提示词。
- 混合村粗：用redis 存储当前对话的数据，保证响应速度，MySQL存储所有历史对话的数据。

## (追问)短期记忆和长期记忆是如何区分和存储的?

### 短期记忆

1. 核心目标：维持当前对话的连贯性
2. 区分：
   （1）只对当前会话有效
   （2）受到大模型单次处理内容的限制
3. 存储
   （1）存储在服务器的内存或者高性能的键值对数据库（redis，临时存储）。可以通过消息队列的方式，将短期记忆数据同步到MySQL中。
   （2）当前上下文窗口

### 长期记忆

1. 核心目标：让AI认识你，实现个性化与持续成长
2. 区分：
   （1）持久化存储
   （2）高度结构化：通过AI把用户的关键信息、偏好、习惯提炼，结构化存储
3. 存储
   （1）向量数据库：将用户信息、偏好、习惯等转化为向量存储，提问时，系统通过语义化检索。
   （2）传统数据库：将明确的信息，包括地址，账号等直接存储

## (追问)记忆更新策略是什么?

1. 冲突合并、删除、跳过
   - 当AI从新对话中提取新的记忆时，先通过语义检索相关部分，如果有则进行：
     （1）合并：将并不冲突的信息进行合并
     （2）删除：如果冲突，则删除
     （3）跳过：如果信息重复，跳过
2. 时间管理
   （1）遗忘曲线与时间衰弱：系统会给记忆打上“热度分”，长期不被访问的记忆会被降低权重或者冷处理（自动归档）
   （2）双时态性：AI记录有效时间与记录时间
3. Git追加式记录：类似Git的追加式记录，保留记忆的完整历史和上下文变更轨迹
4. 记忆操作系统：将记忆分为参数记忆（深层知识，类似长期技能）、激活记忆（工作记忆，类似当前思考）和明文记忆（外部笔记，类似历史记录）。
